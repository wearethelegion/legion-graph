# kgrag-neo4j-storage

Streaming Kafka worker that materialises extracted entities, relationships, chunks, and summaries into the Neo4j knowledge graph.

## What it does

Consumes from three Kafka topics (`extracted-entities`, `text-summaries`, `enriched-code-chunks`) and writes nodes (Entity, EntityType, DocumentChunk) and relationships (LLM-extracted edges, structural edges) to per-company Neo4j databases. Writes are streamed in micro-batches as data arrives; no waiting for pipeline completion. Uses Cognee-compatible schema (APOC-based `__Node__` base label, dynamic labels and relationship types). Filters and canonicalises Business Domain entities against company whitelist from Postgres.

## Where it lives

- **Source:** `graph_services/neo4j_storage_service/`
- **Compose service:** `kgrag-neo4j-storage` (`docker/docker-compose.yml:658-704`)

## Outputs

### Neo4j databases

**Multi-tenant pattern:** Each `company_id` gets an isolated Neo4j database named `cognee-{company_id}`. Database is created automatically via `multi_tenancy.ensure_neo4j_database()` on first write (`main.py:438`).

### Node labels written

All nodes use **Cognee-compatible schema** with APOC dynamic labels:

- **Base label:** `__Node__` (applied to all nodes, enforced by unique constraint on `id`)
- **Secondary labels** (via `apoc.create.addLabels` — `writer.py:382-401`):
  - `Entity` — code/document entities (functions, classes, variables, concepts)
  - `EntityType` — entity type taxonomy (e.g., `function`, `class`, `BusinessDomain`)
  - `DocumentChunk` — code/document chunks (text segments from files or knowledge items)
  - `Repository` — repository metadata nodes (created by hierarchy writer)
  - `Document` — file/document metadata nodes (created by hierarchy writer)

### Relationship types written

**Dynamic relationship types** via `apoc.merge.relationship` (`writer.py:1209-1269`):

| Edge type | Source | Target | Description |
|-----------|--------|--------|-------------|
| LLM-extracted | `Entity` | `Entity` | Relationship types from LLM extraction (e.g., `CALLS`, `INHERITS_FROM`, `DEPENDS_ON`, `USES`). Uppercase by convention. |
| `contains` | `DocumentChunk` | `Entity` | Structural: chunk contains entity |
| `is_a` | `Entity` | `EntityType` | Structural: entity is of type |
| `is_part_of` | `Entity` | `Document` | Structural: entity belongs to document |
| `made_from` | `Document` | `DocumentChunk` | Hierarchy: document composed of chunks |
| `exists_on` | `Repository` | `Branch` | Hierarchy: repository has branch |
| `belongs_to_set` | `Entity`/`EntityType` | `NodeSet` | Scoping: entity belongs to project/company node set |

### MERGE patterns

**All writes use `MERGE` for idempotency** (`writer.py:382-401` for nodes, `writer.py:1209-1269` for edges):

- Nodes: `MERGE (n:__Node__ {id: $id}) ON CREATE SET ... ON MATCH SET updated_at = timestamp()`
- Edges: `MERGE (source)-[r:RELATIONSHIP_TYPE]->(target) ON CREATE SET ... ON MATCH SET updated_at = timestamp()`

Node IDs are UUID5 deterministic (`writer.py:28-35`):
- Entities: `uuid5(NAMESPACE_OID, f"{name}|{node_set}")`
- Chunks: `uuid5(NAMESPACE_OID, chunk_id)`
- Documents: `uuid5(NAMESPACE_OID, f"{project_id}:{file_path}")`

Repeated ingestion of the same data updates timestamps but does not duplicate nodes or edges.

## Dependencies

- **Kafka:** Redpanda (`redpanda:9092`) — upstream topics from `entity_extraction_service`, `summarization_service`, `code_preprocessor`, `document_preprocessor`
- **Neo4j:** DozerDB 5.26.3.0 (`neo4j:7687`) — must support multi-database feature (standard Neo4j Community does not)
- **Postgres:** `kgrag_auth` database — for company business domain whitelist (`code_processing.company_business_domains`); `cognee` database — for Cognee relational tables (`cognee.nodes`, `cognee.edges`)
- **APOC plugin:** Required for dynamic labels and relationship types (`writer.py:329-349`). Env var `NEO4J_dbms_security_procedures_unrestricted=apoc.*` must be set in Neo4j config.

### 1. Start service

```bash
docker-compose up -d kgrag-neo4j-storage
docker logs -f kgrag-neo4j-storage
```

Expected log output:
```

collector.started

worker.started worker_id=1

```

### 2. Verify Kafka consumer is active

```bash
docker exec -it kgrag-redpanda rpk group describe neo4j-storage-streaming
```

Expected: consumer group has 3 topic partitions assigned (`extracted-entities`, `text-summaries`, `enriched-code-chunks`).

### 3. Trigger ingestion

Ingest a small repository via REST API:

```bash
curl -X POST http://localhost:8000/api/v1/code_ingestion \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "repository": "https://github.com/user/repo",
    "branch": "main",
    "framework": "python",
    "project_id": "your-project-uuid"
  }'
```

Or add a document via brain API:

```bash
curl -X POST http://localhost:8000/api/v1/brain \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "kind": 2,
    "title": "Test Knowledge",
    "content": "This is test knowledge content."
  }'
```

### 4. Verify nodes are appearing in Neo4j

Open Neo4j Browser at `http://localhost:7474` (credentials: `neo4j` / `$NEO4J_PASSWORD`).

Switch to the correct company database:

```cypher
:use cognee-<company_id>;
```

**Query 1: Check base label and node types**

```cypher
MATCH (n:__Node__)
RETURN labels(n) AS node_labels, count(*) AS count
ORDER BY count DESC
LIMIT 10;
```

Expected: rows showing `[__Node__, Entity]`, `[__Node__, EntityType]`, `[__Node__, DocumentChunk]`.

**Query 2: Verify entities from a specific file**

```cypher
MATCH (chunk:__Node__:DocumentChunk)-[:contains]->(entity:__Node__:Entity)
WHERE chunk.file_path CONTAINS 'main.py'
RETURN entity.name, entity.entity_type, entity.description
LIMIT 20;
```

Expected: entities extracted from `main.py` (functions, classes, variables).

**Query 3: Verify entity type distribution**

```cypher
MATCH (e:__Node__:Entity)-[:is_a]->(t:__Node__:EntityType)
RETURN t.name AS entity_type, count(e) AS entity_count
ORDER BY entity_count DESC;
```

Expected: counts for `function`, `class`, `variable`, `BusinessDomain`, etc.

**Query 4: Verify LLM-extracted edges**

```cypher
MATCH (a:__Node__:Entity)-[r]->(b:__Node__:Entity)
WHERE type(r) <> 'is_a' AND type(r) <> 'is_part_of'
RETURN type(r) AS relationship_type, count(*) AS count
ORDER BY count DESC
LIMIT 10;
```

Expected: relationship types like `CALLS`, `USES`, `INHERITS_FROM`, `DEPENDS_ON`.

**Query 5: Check constraint is in place**

```cypher
SHOW CONSTRAINTS;
```

Expected: unique constraint on `__Node__.id` (created by `writer.ensure_constraints()` — `writer.py:329-349`).

## Operational notes

### Neo4j database creation

**The `cognee-{company_id}` database must exist before writes.** Service creates it automatically on first batch for each company via `multi_tenancy.ensure_neo4j_database()` (`multi_tenancy.py:32-70`). This issues:

```cypher
CREATE DATABASE `cognee-{company_id}` IF NOT EXISTS
```

against the `system` database. DozerDB supports `CREATE DATABASE` in Community Edition; standard Neo4j Community does not. If using standard Neo4j, you must manually create databases or use Enterprise.

### MERGE idempotency

All writes use `MERGE` (`ON CREATE SET ... ON MATCH SET updated_at = timestamp()`). Repeated ingestion of the same data is safe and updates timestamps without duplication. This enables:
- Forced full refresh (`force_full_refresh=true` in ingestion request)
- Re-running failed ingestions
- Incremental updates via webhook-triggered ingestion

**Caveat:** If entity name or file path changes, a new node is created (UUID5 is name-based). Old nodes are not automatically deleted.

### Batch size tuning

Two batch sizes control throughput:

1. **Micro-batch size** (`NEO4J_STORAGE_BATCH_SIZE`, default 100): events per batch collected from Kafka before writing to Neo4j. Larger = better throughput, higher latency. Smaller = lower latency, more Neo4j transactions.

2. **UNWIND batch size** (`NEO4J_BATCH_SIZE`, default 500): rows per UNWIND batch in Cypher writes (`writer.py:362-373`). Larger = fewer round-trips to Neo4j, more memory per transaction. Smaller = lower memory, more queries.

Recommended starting points:
- Default (100 / 500): balanced for most workloads
- High throughput (200 / 1000): more aggressive batching
- Low memory (50 / 250): conservative for constrained environments

### Business Domain canonicalisation

When `ENABLE_DOCUMENT_CANONICALISATION=true` (default), the service loads the company's Business Domain whitelist from `code_processing.company_business_domains` table (`main.py:130-168`). Entities with `entity_type="BusinessDomain"` are validated against this whitelist:

- **Whitelist present:** Only canonical values are allowed. Casing/punctuation drift is force-corrected. Entities with invalid values are rejected and never written to Neo4j.
- **Whitelist empty:** No enforcement (legacy behaviour, all values accepted).

Whitelist is cached per company for the lifetime of the service process. To refresh, restart the service.

Canonicalisation code: `shared.canonicaliser.canonicalise_entity()` (called from `main.py:647-649`).

### Dual-write to Cognee relational tables

In addition to native Neo4j writes, the service also writes nodes and edges to Cognee's relational Postgres tables (`cognee.nodes`, `cognee.edges`) via `cognee.modules.graph.methods.upsert_nodes()` and `upsert_edges()` (`main.py:978-1126`). This ensures both search paths (Cognee gRPC search via `kgrag-cognee:50052` and direct Neo4j/Qdrant search via `kgrag-search:50051`) see the same data.

### Graceful shutdown

Service handles SIGINT/SIGTERM gracefully (`main.py:1571-1612`):
1. Stop accepting new Kafka messages
2. Drain work queue into final batch
3. Wait for collector and workers to finish
4. Commit Kafka offsets
5. Close Neo4j driver and Postgres pool

Use `docker stop kgrag-neo4j-storage` (default 10s timeout) for graceful shutdown.

### Monitoring

**Service logs:**

```bash
docker logs -f kgrag-neo4j-storage
```

Key log events:
- `service.starting` — startup config
- `collector.batch_pushed` — micro-batch sent to workers
- `worker.batch_complete` — batch write finished (nodes/edges counts, duration)
- `worker.batch_error` — batch write failed (error details)

**Neo4j query log:**

```bash
docker exec -it kgrag-neo4j tail -f /logs/query.log
```

Shows Cypher queries executed by the service.

**Kafka consumer lag:**

```bash
docker exec -it kgrag-redpanda rpk group describe neo4j-storage-streaming
```

Shows offset lag for each topic partition. High lag indicates service is falling behind ingestion rate.
